# Move LLM reason-code narration OFF the Databricks cluster and ONTO FastAPI

> Archived verbatim (governance requirement). The exact task that produced the off-cluster
> narration change (engine flag + `narration_context.json` side artifact + the
> `POST /api/v1/runs/{run_id}/narrate` endpoint + the frontend auto-narrate + removal of the
> now-moot secret-scope handoff). Supersedes `databricks_llm_narratives_secret_scope.md`.

---

ClassifyOS is a GenAI-developed ML classification framework for the insurance domain: it predicts
categorical outcomes (lapse, fraud, risk tier, etc.) from tabular data. Three layers — React
frontend → FastAPI backend → pure-Python ML engine. The engine cleans, engineers features, balances
classes, trains and compares multiple models (with optional Optuna tuning), and reports the best one;
results surface in a dashboard.

Before doing anything, read PROJECT_STATE.md and the relevant *_short_desc.md for the surface(s)
you're touching (backend_short_desc.md / api_short_desc.md / frontend_short_desc.md), plus CLAUDE.md
for the hard rules. Also check bugs.md and docs/databricks_integration.md if the task relates to known
issues or the Databricks/MLflow roadmap.

Respect the project's constraints: no data leakage (fit on train only); additive changes only (new
capabilities behind flags); all file I/O through StorageAdapter (no hardcoded paths); the /api/v1/run
contract is LOCKED — additive-only with a schema_version bump, and update docs/api_contract.md. New
optional dependencies follow the opt-in / lazy-import / report-only discipline of shap/optuna/mlflow
(OFF by default, imported only when used, never abort a run on failure).

My task:

Move LLM reason-code narrative generation (Azure OpenAI; schema 1.7 —
result.explanations[model].rows[].narrative) OFF the Databricks cluster and ONTO FastAPI, with FULL
context parity. Background: narratives are generated by the engine DURING the run
(classifyos.analysis.llm_explain via ModelRunner). On Databricks the run executes on the CLUSTER, which
CANNOT reach the Azure OpenAI endpoint — it is locked to a private endpoint, so every call fails with
"openai.PermissionDeniedError: 403 - Public access is disabled. Please configure private endpoint."
(confirmed in a live job log). FastAPI CAN reach the endpoint (that is why local works). The prior
secret-scope fix correctly delivered the creds to the cluster, but the cluster still can't reach Azure
— so narration must move to FastAPI.

Design (agreed):
- ENGINE (wheel change; additive + flag-gated): keep computing SHAP. NEW: serialize the RunContext the
  runner already builds (ModelRunner._build_run_context) to a SIDE artifact api/narration_context.json
  — carrying the fields the API can't otherwise get: dataset_context, column_context, context_mode
  (from the run config) + derived_schema + sample_rows (data-derived). Gate the in-engine LLM narrate
  call behind a new flag CLASSIFYOS_NARRATE_IN_ENGINE (default TRUE → local unchanged/byte-identical).
  It is a SIDE ARTIFACT, not part of the locked /run envelope → NO schema_version bump.
- NOTEBOOK (notebooks/classifyos_job_runner.py, Cell 3): set CLASSIFYOS_NARRATE_IN_ENGINE=false so the
  cluster skips the unreachable Azure call and just writes SHAP + narration_context.json. Also disable
  MLflow OpenAI autologging (the source of the "experiment_id is missing" /
  "/Users/.../classifyos_job_runner does not exist" warnings + the added latency). The azure_secret_scope
  forwarding from the previous change is now MOOT (the cluster no longer narrates) — stop forwarding it
  (leave sync_llm_secrets dormant or remove it; your call, keep it clean).
- FASTAPI: new endpoint POST /api/v1/runs/{run_id}/narrate. It loads the run's envelope
  (api/mlflow_read.load_run) + the narration_context.json artifact from the same MLflow run, rebuilds
  the FULL RunContext (dataset/column/context_mode + derived_schema + sample_rows from the artifact;
  class_base_rates from result.run.class_distribution; per-model metrics from result.models;
  global_features from result.feature_importance/permutation_importance; SHAP rows from
  result.explanations), builds the narrator via classifyos.analysis.llm_explain.narrator_from_env
  (FastAPI's AZURE_OPEN_AI_* creds), computes the primer via derive_dataset_understanding, narrates
  every row via narrate_rows, attaches result.explanations[...].rows[].narrative, and PERSISTS the
  narrated envelope back as the run's snapshot (routed to Databricks MLflow with
  tracking_uri="databricks", mirroring the §6.1 read fix) so a reloaded run shows narratives instantly.
  Report-only: absent creds / absent context / any failure → return the envelope unchanged, never 500.
  Reuse classifyos.analysis.llm_explain entirely (it's already importable in the API env — no new engine
  code for the narrator itself).
- FRONTEND: on the Explainability page for a databricks-backed run (or when such a run loads), if
  result.explanations is present but rows lack narrative, call /runs/{run_id}/narrate and swap in the
  narrated envelope. For the demo the run is pre-narrated once (persisted), so the reloaded past run is
  instant.

Constraints/notes: LOCAL backend stays byte-identical (CLASSIFYOS_NARRATE_IN_ENGINE defaults true →
engine narrates in-process as today; the extra context artifact is harmless). No /run envelope shape
change (narration_context is a side artifact; narrative already exists since 1.7) → NO schema_version
bump. This DOES require a wheel rebuild + cluster restart + notebook re-import (B-full). Narratives are
baked into a run's stored snapshot at narrate time — existing past runs stay null until re-narrated.
Verify end-to-end where sensible; all Azure/MLflow/Databricks MUST be mocked in tests (reuse

## Implementation notes (as built)

- The runner had no literal `_build_run_context`; the `RunContext` was assembled inline in
  `ModelRunner._add_llm_narratives`. A NEW `ModelRunner._build_narration_context` was added (kept
  SEPARATE from `_add_llm_narratives` so the in-engine local path stays byte-identical), returning only
  the fields the envelope can't provide (context_mode / dataset_context / column_context / derived_schema
  / sample_rows / feature_cols), JSON-sanitized via `envelope.serialize.safe_jsonify`.
- The side artifact is logged inside `mlflow_logging.log_run` (new `narration_context` param) under the
  `api/` MLflow subdir (constant `NARRATION_CONTEXT_PATH`), next to the result snapshot — NOT written to
  OUTPUT_DIR, so the local output folder stays byte-identical too.
- `snapshot_envelope` gained an optional keyword `tracking_uri`; `api.mlflow_read.snapshot_result` passes
  `_tracking_uri()` so the narrate step's re-persist targets Databricks-managed MLflow per call.
- The API narrate logic lives in `api/narrate.py` (`narrate_envelope`), which reuses the engine narrator
  entirely and rebuilds `original_row` from the envelope's `feature_values` (`_reconstruct_original_row`,
  longest-prefix source-column mapping) for prompt-value parity.
- The secret-scope path was REMOVED (not left dormant): `sync_llm_secrets` + Secrets helpers, the
  `azure_secret_scope` submit parameter/base_parameter, and the notebook widget/loop. The historical
  secret-scope prompt (`databricks_llm_narratives_secret_scope.md`) is kept as the record.
- Frontend: `narrateRun` client fn + a `useAutoNarrate` effect on Explainability (once per run id,
  databricks-backed only, skips when narratives already present).
