My task:

Make the opt-in LLM reason-code narratives (Azure OpenAI; schema 1.7 — result.explanations[model].rows[].narrative)
work when a run executes on the DATABRICKS execution backend. They work locally but are always null on Databricks.

Root cause (already confirmed): the narrative is generated by the engine DURING the run (ModelRunner ->
classifyos.analysis.llm_explain.narrator_from_env), which reads the five AZURE_OPEN_AI_* creds from os.environ. On
Databricks the run executes on the CLUSTER, whose environment has no AZURE_OPEN_AI_* (the Job notebook Cell 3 sets
storage/MLflow/token vars but not these), so narrator_from_env() returns None and logs "LLM narratives requested but
these env vars are unset: AZURE_OPEN_AI_ENDPOINT, AZURE_OPEN_AI_API_KEY, AZURE_OPEN_AI_API_VERSION,
AZURE_OPEN_AI_DEPLOYMENT_NAME; shipping SHAP only" (verified in a live Databricks job log). Locally the FastAPI process
has them in .env. The openai package is already installed on the cluster (a wheel dependency) — it is purely the
missing credentials.

Fix (robust — route the creds through a Databricks SECRET SCOPE; the Azure key must NEVER land in the Job's run
parameters or logs):
- FastAPI (databricks submit branch only): when AZURE_OPEN_AI_* are present in the FastAPI env, ensure a secret scope
  exists (default name "classifyos-llm", overridable via a new env var e.g. CLASSIFYOS_DBRICKS_SECRET_SCOPE) and write
  the 5 creds into it via the Databricks Secrets REST API — secrets/scopes/create with initial_manage_principal="users"
  (so any workspace identity, including whatever the cluster runs as, can read it — the DATABRICKS_TOKEN is a workspace
  USER token, sharmil.basa@sapiens.com, verified able to list/create scopes), then secrets/put per key — authenticated
  with the service token via the existing _build_client/_request seam. Idempotent + report-only: any failure just means
  no narratives, never block the submit. Pass ONLY the scope name to the Job as a base_parameter (non-secret) — never
  the values.
- Notebook (notebooks/classifyos_job_runner.py, Cell 3): when the scope-name parameter is present, read each of the 5
  creds via dbutils.secrets.get(scope, key) and set os.environ[...] BEFORE ModelRunner.run(); guard with try/except
  (report-only) so a missing scope/secret just leaves narratives off. Databricks redacts secret values in notebook output.
- Opt-in / gating: no creds in the FastAPI env -> nothing synced or forwarded -> narratives off, no error, byte-identical
  to today. LOCAL backend unchanged (in-process narrator reads .env directly). Additive: the /run envelope already carries
  `narrative` (since 1.7), so NO schema_version bump.

Constraints to honor: the whole point is that the Azure key stays in .env + the secret scope only (not in Job params —
that was the naive approach's flaw). Document the initial_manage_principal="users" choice (scope readable by any
workspace user) and note a tighter per-principal ACL (secrets/acls/put granting READ to the cluster's run-as identity)
as a follow-up. No engine/wheel change (llm_explain + runner are unchanged — they just find the env vars now); deploy =
FastAPI change + notebook re-import + FastAPI restart, no cluster restart.

Demo note (so the fix is verifiable): narratives are baked into a run's stored envelope AT RUN TIME, so existing past
runs stay null on reload — after deploying, execute ONE run so it logs narratives; THAT run is the one to reload in the
Runs tab.

## What was implemented (summary)

- **`api/databricks.py`** — Secrets client: `sync_llm_secrets()` (returns the scope name on success, else None;
  report-only), `_create_secret_scope` (idempotent, tolerates `RESOURCE_ALREADY_EXISTS`), `_azure_llm_creds` (reuses
  `llm_explain`'s ENV constants — single source), `llm_secret_scope()` / `_llm_secret_read_principal()` (env-overridable,
  defaults `classifyos-llm` / `AIML_RD`). Paths: `secrets/scopes/create`, `secrets/put`, `secrets/acls/put`. Also added
  `azure_secret_scope` to `_submit_payload`/`submit_run` (base_parameter — the scope NAME only).
- **`api/routes/run.py`** — databricks submit branch: when `cfg.explainability.llm_narratives` and creds present, sync
  (threadpooled) and forward the returned scope name to `submit_run`.
- **`notebooks/classifyos_job_runner.py`** — Cell 2 `azure_secret_scope` widget; Cell 3 reads each cred via
  `dbutils.secrets.get` into `os.environ` before `ModelRunner.run()` (report-only).
- **Tests (`test_api_jobs.py` +6, all REST mocked):** sync pushes creds + grants READ; absent-creds no-op; tolerates an
  existing scope; submit forwards the scope name only when narratives+creds present (key NOT in params); empty when
  narratives off / creds absent.
- **Docs:** `databricks_how_it_works.md` §15, `databricks_wisdom.md` (playbook row), `api_short_desc.md`, `.env.example`
  (`CLASSIFYOS_DBRICKS_SECRET_*`), `PROJECT_STATE.md`. No `schema_version` bump. plan_tweak untouched (no deviation).
